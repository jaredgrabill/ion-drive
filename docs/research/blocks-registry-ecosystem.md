# Research: Block Distribution Ecosystems (shadcn, Helm, npm/sigstore)

**Date:** 2026-07-08. **Feeds:** ADR-022 and the spec suite at
`docs/specs/blocks-ecosystem/`. **Question:** how should Ion Drive distribute blocks —
main hosted registry, third-party self-hosted registries, CLI publishing, versioning,
and integrity — learning from shadcn and the registries that solved the problems shadcn
hasn't?

## 1. The shadcn registry ecosystem (primary model)

**What a shadcn registry is:** static JSON over HTTPS. A `registry.json` index (name,
homepage, `items[]` summaries) plus one JSON file per item embedding the actual file
contents (`registry-item.json` schema: name, type — `registry:block|component|ui|hook|
lib|style|theme|…` — title, description, `files[{path,type,target,content}]`,
`dependencies` (npm), `registryDependencies` (other items), cssVars, envVars, docs,
categories, meta). No server. Anyone with a static host has a registry. This is the
single biggest reason the ecosystem exploded — the barrier to *hosting* is zero.

**Namespaced registries (CLI 3.0, Aug 2025):** `components.json` gains
`registries: { "@acme": "https://…/{name}.json" | { url, headers, params } }` with
`${ENV_VAR}` expansion for private registries (bearer/API-key/query-param auth all in
the wild). Items reference cross-registry deps as `@acme/input`. The resolver fetches
recursively, topo-sorts, dedupes by target path (last-wins — deliberate override
mechanic), detects cycles, and keeps per-registry auth contexts.

**Authoring/publishing:** `shadcn build` compiles a source registry (optionally composed
from multiple files via `include`, May 2026) into flat per-item artifacts;
`shadcn registry validate` lints before publishing; `shadcn/registry` exports loaders
for dynamic routes. Publishing is "put the files somewhere" — there is no publish
protocol.

**Discovery/trust:** a PR-reviewed central directory (`ui.shadcn.com/r/registries.json`;
requirements: open-source, schema-conformant, flat layout, no embedded content in the
listed index) consulted by `shadcn search`. Review is manual and listing-level. An MCP
server exposes registries to coding agents.

**What works (we adopt):** static-file protocol; namespace map with env-expanded auth;
cross-registry deps with recursive resolution; build/validate authoring toolchain;
PR-reviewed directory; MCP for agents; "code lands in your tree" ownership (we already
adopted this in ADR-018).

**What's missing (we must not copy):**
- **No versioning.** Items are latest-only; "versioning" is a query-param convention.
  No immutability, no lockfile, no update mechanism beyond `diff` against latest.
- **No integrity.** No hashes, signatures, or provenance anywhere. Acceptable for UI
  code a developer reads in their editor before shipping; unacceptable for Ion Drive
  blocks, which execute **server-side DDL, seed data, register webhook handlers, and
  vendor code that runs in the user's backend**. Our stakes are npm-package stakes,
  not copy-paste-snippet stakes.
- No publisher identity or trust marks.

## 2. Helm classic chart repositories (the structural precedent for versioned static)

A Helm repo is an HTTP server with an `index.yaml` listing every chart's every version,
each entry carrying metadata + a **`digest`**; packaged charts and **provenance files at
an adjacent URI** (`mychart-1.2.3.tgz.prov` — chart hash + PGP signature; `helm verify`
/ `install --verify` checks both). Proven pattern: *versioned immutable artifacts +
digest-in-index + provenance-adjacent + still just static files*.

Helm's known failure mode: the **monolithic index** — big repos ship multi-MB
`index.yaml` downloads to install one chart. Their fix was migrating to OCI registries
(content-addressable, auth built-in, per-chart pulls). Lesson for us: split the index
(small directory file + per-block version files) from day one; note OCI as a future
distribution option (an `artifactUrl` can point anywhere, so this needs no protocol
change), don't build it now.

## 3. npm trusted publishing + sigstore (the provenance/trust model)

npm provenance implements SLSA via sigstore: the GitHub Actions job's **OIDC token** →
**Fulcio** issues a short-lived X.509 cert binding the identity (repo, workflow, sha) →
the artifact digest is signed → logged in **Rekor**, a public transparency log. With
**trusted publishing**, the registry accepts the OIDC token directly — no long-lived
secret exists at all — and provenance is generated by default; npm displays a
provenance badge. GitHub generalized this as **artifact attestations**
(`actions/attest-build-provenance` + `gh attestation verify`) for *any* artifact in any
repo — which means we get npm-grade provenance for block artifacts **without running any
infrastructure or PKI**: attest in the publish workflow, commit the sigstore bundle next
to the artifact, verify client-side with the `sigstore` npm library against the public
trust root.

Ion Drive already uses the consumer side of this: `release.yml` publishes all seven npm
packages via OIDC trusted publishing with automatic provenance. The blocks pipeline
reuses the exact discipline (tag/main-gated, guards, dry-run rehearsal,
skip-if-published idempotency).

## 4. Other reference points (briefly)

- **Terraform Registry:** protocol-over-discovery (`.well-known/terraform.json`) —
  federated by design; validates our "protocol first, hosts are interchangeable" stance.
- **JSR:** scoped packages + provenance-by-default from day one — proof that a new
  registry can make attestation the norm rather than an add-on; we start attested too.
- **VS Code Marketplace / Open VSX:** verified-publisher = domain/org control checks;
  their malware incidents show marketplace curation is an ongoing operational duty —
  hence our advisory/yank/takedown machinery is specced from the start, not bolted on.
- **TUF (The Update Framework):** the rigorous answer to *registry-file* (index)
  compromise. Deliberately **not adopted** for M1–M2: our index rides HTTPS + digest
  pinning + attestation cross-checks + consumer-side recorded digests (mutation is
  loudly detectable); TUF's key-management burden is disproportionate for a
  solo-maintained git-hosted registry. Recorded as the far-future option if the hosted
  registry ever needs offline-verifiable index integrity.

## 5. Our problem is different from shadcn's (backend, not frontend)

1. **Installing a block mutates a live system** (DDL, seeds, roles, scheduled tasks,
   webhook endpoints). Integrity failures are not "ugly button" failures. ⇒ digests are
   mandatory, verification is not skippable, and destructive changes stay behind the
   existing preview/force machinery.
2. **Blocks are singletons per server** (one `crm`, one ledger row, one
   `/blocks/crm` dir). ⇒ semver ranges are *compatibility constraints*, not an
   npm-style resolution problem; the project config can double as the lockfile.
3. **Two-part install** (vendored code + server manifest) means two integrity surfaces
   with one artifact — hash the artifact, and both are covered, *if* verification
   happens before vendoring and before POSTing. ⇒ verify-then-parse ordering is load-
   bearing.
4. **The server keeps a manifest snapshot ledger** — richer than anything shadcn has;
   it powers `diff`/`update` (three-way against pristine bytes) and post-incident
   forensics ("which servers installed the bad digest").
5. **Agents are first-class users** — registry search/preview must be MCP tools, and
   `--json` everywhere, because the "developer" running `add` is often an LLM.

## 6. Decision summary (details in ADR-022 + specs)

Adopt shadcn's shape (static files, namespaces, env-auth, directory, build/validate,
MCP), Helm's versioned-artifact discipline (split index, per-version digests,
provenance-adjacent, immutability), and npm/sigstore's trust machinery (trusted-
publishing-style CI flow, attestation-derived `verified` mark, transparency log) — and
add the pieces our backend posture demands (hard-fail digest verification, install
provenance in the server ledger, advisory/yank protocol, upgrade mode behind
preview-first). Milestones M1 (protocol+trust+publish, all static) → M1.5 (SDLC:
`block test`, `diff`/`update`) → M2 (site/search/directory/MCP) → M3 (hosted write API
on Ion Drive itself; read path static forever).
